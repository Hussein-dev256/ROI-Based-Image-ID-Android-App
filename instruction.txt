## 1. The Problem (Business + Product Perspective)

Right now, **ObjectID exists only as an APK**.

From a **user / recruiter / client** perspective:

* They cannot preview the product instantly
* They must trust and install an APK (high friction, low trust)
* They cannot experience ObjectID alongside your other live projects
* Your strongest work looks *less impressive* than weaker projects that have live demos

From a **portfolio conversion** perspective:

* APK-only projects underperform
* Visitors bounce instead of engaging
* You lose credibility despite solid engineering

üëâ **The problem is not ObjectID.
The problem is how it is presented.**

---

## 2. Product Goal (What We Are Fixing)

**Primary Goal**

> Make ObjectID instantly usable from the browser with zero installation.

**Secondary Goals**

* Keep the existing Android app untouched
* Avoid re-training models
* Minimize cost and hosting dependencies
* Make ObjectID look like a *real product*, not a student demo

**Non-Goals**

* We are NOT replacing the Android app
* We are NOT rebuilding the entire system
* We are NOT chasing perfect UI before functionality

---

## 5. Developer Execution Plan (Step-by-Step)

### Phase 1 ‚Äî Stabilize the Core (DO NOT SKIP)

**Objective**
Ensure your current inference logic is stable and reusable.

**Tasks**

* Extract image preprocessing into a standalone function
* Extract model loading into a singleton/service
* Define a single inference function:

```text
input: image
output: { label, confidence[] }
```

**Rules**

* No UI work yet
* No optimization yet
* No refactoring unrelated code

**Acceptance Criteria**

* Given the same image, APK and API return the same results

---

### Phase 2 ‚Äî Build the Inference API

**Objective**
Expose ObjectID as a service.

**Endpoint**

```
POST /identify
Content-Type: multipart/form-data
```

**Request**

* image file
* optional flags (top_k, threshold)

**Response**

```json
{
  "predictions": [
    { "label": "Dog", "confidence": 0.92 }
  ]
}
```

**Non-Negotiables**

* Input validation
* File size limits
* Timeouts
* Error handling (no stack traces to client)

---

## 6. Risk Management (How We Avoid Breaking Things)

### Rule #1 ‚Äî APK is Sacred

* No changes to the existing Android code
* No shared mutable code
* No forced refactors

---

### Rule #2 ‚Äî Feature Flags

* Web demo uses its own config
* Separate model path if needed
* Environment-based switching

---

### Rule #3 ‚Äî Validation Parity

Before deployment:

* Test same image on:

  * Android app
  * API
* Compare labels and confidence ranges

---

### Rule #4 ‚Äî Performance Guardrails

* Max image size enforced
* CPU inference only
* Timeout hard limit (e.g. 5s)
